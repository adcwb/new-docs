# CLAUDE.md — 博客文档写作规范

本项目是基于 **Hugo + Hextra 主题** 的技术博客 / 文档站。所有文章位于 [content/docs/](content/docs/)，按分类目录组织。本文件定义统一的 Markdown 写作风格，**新增或修改文档时必须遵循**。

站点同时支持中文（默认）和英文，文件命名规则：中文版 `filename.md`，英文版 `filename.en.md`。

---

## 1. 文件与目录组织

- 所有文章放在 `content/docs/<分类>/...`，分类目录即侧边栏一级菜单。
- 每个目录必须有一个 `_index.md`（中文）和 `_index.en.md`（英文），定义该分类的 `title` 和 `weight`。
- 文件名使用有意义的名称；同一目录内若有先后顺序，用前缀数字（如 `01`、`1.`）表达，并在 front matter 用 `weight` 固化顺序。
- 资源（图片等）就近存放，优先使用页面 bundle：`my-post/index.md` + `my-post/img.png`，正文用相对路径 `![](img.png)`。

## 2. Front Matter（必填）

每篇文档**必须**以 YAML front matter 开头：

```yaml
---
title: "文章标题"        # 显示在侧边栏、面包屑、浏览器标签
weight: 10              # 同级排序，数字越小越靠前
date: 2026-06-05        # 发布日期
tags: ["Go", "并发"]    # 至少 1~3 个标签，与内容主题匹配
# 可选项：
# draft: false          # true 时不会被构建
# math: true            # 需要 KaTeX/MathJax 公式时开启
---
```

约定：

- `title` 一律用双引号包裹，避免以数字、冒号、特殊符号开头时的 YAML 解析问题。
- `title` 不重复目录名，不带排序数字前缀（排序交给 `weight`）。
- `tags` 为必填项，至少写 1 个，便于站内检索和分类。
- 上线内容确保没有 `draft: true`。

## 3. 标题层级

- front matter 的 `title` 即页面 H1，**正文不要再写重复的一级标题**。
- 正文从 `##`（H2）开始，逐级 `###`、`####`，不可跳级。
- **禁止使用 `1.1`、`1.2` 等带编号的标题格式**，统一用 Markdown 标题层级表达结构。
- Hextra 会根据正文标题自动生成右侧目录（TOC）。

## 4. 禁止 Typora 专有写法

本站由 Hugo（Goldmark）渲染，以下 Typora 习惯会导致显示异常，**禁止使用**：

- ❌ `[toc]` / `[TOC]` —— Hextra 自动生成目录，写了会渲染成字面文字。
- ❌ 行首用全角空格（`　`）或零宽空格（U+200B）做缩进 —— 会造成排版错乱甚至误判为代码块。段落首行不缩进，靠空行分段。
- ❌ 行内 `$...$` 数学公式默认不开启，需要时在 front matter 加 `math: true`。
- ❌ `>文字`（`>` 后无空格）—— 必须写成 `> 文字`，且 `>` 仅用于真正的引用或注解，不作缩进排版。

## 5. 代码块

- 必须标注语言，便于高亮：

  ````markdown
  ```go
  func main() {}
  ```
  ````

- 语言标签要与内容匹配：
  - Shell 脚本 → `bash`
  - SQL → `sql`
  - 配置文件 → `ini` / `yaml` / `toml`
  - 公式、伪代码、纯文本说明 → `text`
  - LaTeX 示例 → `latex`
- 不要把 SQL/Shell 误标成 `python`，不要使用无标注的 ` ``` ` 开头块（即使是纯文本也要标 `text`）。
- 终端示例可保留提示符（`$`、`#`），但不要把多语义内容塞进一个代码块。

## 6. Hextra 增强组件（Shortcodes）

适当使用 Hextra 提供的短代码提升可读性：

- **提示框**：`{{</* callout type="info" */>}}` 内容 `{{</* /callout */>}}`（type 可为 `info` / `warning` / `error`）。
- **折叠详情**：`{{</* details title="点击展开" */>}}` … `{{</* /details */>}}`。
- **标签页**：`{{</* tabs */>}}{{</* tab name="Go" */>}}` … `{{</* /tab */>}}{{</* /tabs */>}}`（每个 `tab` 上用 `name` 指定标题；旧的 `tabs items="..."` 写法已废弃，**禁止使用**）。
- **步骤**：`{{</* steps */>}}` … `{{</* /steps */>}}`。

普通文章用标准 Markdown 即可，组件按需点缀，不要滥用。

## 7. 其它排版约定

- **列表**：无序用 `-`，有序用 `1.`；同级缩进 2 个空格。
- **表格**：用标准 Markdown 表格，表头与分隔行对齐。
- **引用**：`>` 后加空格（`> 文字`）用于真正的引用/注解，不要拿它做缩进排版。
- **链接**：站内引用其它文档使用相对路径；外链注明来源。
- **强调**：术语首次出现可加粗；避免整段加粗。
- 中英文之间、中文与数字之间建议留一个空格，提升可读性。

## 8. 写作语气与内容结构

参考标准：`人工智能/**`、`金融相关/**`、`PHP开发/**` 等分类的文章为本站写作质量基准。

- 面向普通用户和 IT 从业者，简洁直接，先结论后展开，最后举例验证。
- **每篇正文开头必须有 1~2 段导言**，说明「这篇讲什么、解决什么问题、技术实现原理等」，不要直接从 `##` 标题开始。
- 命令、配置给出可复制的完整示例，并说明关键参数含义。
- 长文用二级标题切分主题，方便 TOC 导航。
- 新增或修改技术类章节时，使用 Context7 搜索该技术的最新文档，确保内容与当前版本一致；涉及多语言/多版本时，明确指出语法差异。
- 内容较少的章节，根据现有内容由浅入深补充子篇，确保知识链完整。

## 9. 多语言维护规则

本站同时维护中文版（`.md`）和英文版（`.en.md`）。

**每次修改或新增中文文章，必须同步更新对应的英文版文件。**

- 若英文版尚未翻译，保留占位符模板（见下），并更新其 `title` 和 `weight` 与中文版保持一致。
- 若英文版已有内容，同步翻译修改的章节，保持两版本内容一致。
- 占位符模板：

  ```markdown
  ---
  title: "English Title"
  weight: 10
  date: 2026-06-05
  tags: ["tag1", "tag2"]
  ---

  {{</* callout type="info" */>}}
  This article has not been translated to English yet. Use the language toggle in the top navigation bar to switch to Chinese and read the full content.
  {{</* /callout */>}}
  ```

- 英文版 `title` 应为对应中文标题的英文翻译，**不要出现中英文混拼**（如 `"大语言Models & Patterns入门"` 是错误示范）。
