广域网用串行链路提供远距离数据传输。这篇讲两种典型串口封装协议 **HDLC** 和 **PPP**、宽带接入常用的 **PPPoE**，以及用于快速故障检测的 **BFD**。

## 串行链路的数据传输方式

串行链路普遍用于广域网，定义了两种数据传输方式：

- **异步传输**：以字节为单位，每个字节用起始位（二进制 0）和停止位（二进制 1）标记开始与结束。起始/停止位占比大，每字节都有额外开销。
- **同步传输**：以帧为单位，通信时用时钟同步两端设备。

### DTE 与 DCE

| 角色 | 全称 | 职责 |
| --- | --- | --- |
| DTE | Data Terminal Equipment（数据终端设备） | 产生/接收数据，如计算机、打印机；通常是公头（针头） |
| DCE | Data Communications Equipment（数据通信设备） | 提供时钟、信号变换和编码，负责建立/保持/释放链路；通常是母头（孔头） |

DTE 产生的二进制数据不适合直接远距离传输，引入 DCE 进行调制/转换。路由器通常作为 DTE，modem、协议转换器等传输设备通常作为 DCE。

> 例：一台位于网络边缘的路由器，其 S0 口要从对端学习参数（如时钟速率），就不需配置时钟速率，它是 DTE，对端是 DCE。

## HDLC

**HDLC（High-Level Data Link Control，高级数据链路控制）**是一种**面向比特**的链路层协议，由 ISO 制定，传送的信息单位为**帧**。

特点：

- 不依赖任何字符编码集。
- 数据报文可透明传输（用"0 比特插入法"，易于硬件实现）。
- 全双工通信，不必等待确认即可连续发送，传输效率高。
- 所有帧采用 CRC 校验，并对信息帧编号，防止漏收或重收，可靠性高。
- 传输控制与处理功能分离，灵活性强。

### 帧结构

完整 HDLC 帧由以下字段组成：

| 字段 | 说明 |
| --- | --- |
| 标志字段（F） | `01111110`，标志帧的开始与结束，也可作帧间填充 |
| 地址字段（A） | 携带地址信息 |
| 控制字段（C） | 构成各种命令与响应，对链路进行监视与控制 |
| 信息字段（I） | 任意长度二进制数，上限由 FCS 或缓存决定，下限可为 0 |
| 帧校验序列（FCS） | 用 16 位 CRC 对两标志字段之间内容校验 |

三种帧类型：

- **信息帧（I 帧）**：传送有效信息或数据。
- **监控帧（S 帧）**：用于差错控制和流量控制，控制字段前两位为 `10`，无信息字段，固定 6 字节。
- **无编号帧（U 帧）**：用于链路的建立、拆除及多种控制功能。

### 基本配置

```text
link-protocol hdlc        # 使能接口的 HDLC 协议
```

{{< callout type="info" >}}
华为设备串行接口**默认运行 PPP 协议**。串行链路两端必须配置相同的链路协议才能通信。
{{< /callout >}}

**HDLC 接口地址借用**：一个没有 IP 地址的接口无法生成路由。IP 地址借用允许无地址接口从其它接口借用 IP，节省地址资源。一般建议借用 loopback 接口的地址，因为它总是 active 状态，能提供稳定可用的 IP。验证命令 `display ip interface brief`：若借用成功，同一 IP 会显示在多个接口上。

## PPP

**PPP（Point to Point Protocol，点对点协议）**是一种点到点链路层协议，主要用于在全双工的同步/异步链路上进行点到点数据传输，提供良好的访问控制和计费功能。

## PPPoE

**PPPoE（PPP over Ethernet）**通过在以太网上提供点到点连接、建立 PPP 会话，使以太网中的主机能连接到远端宽带接入服务器。它兼具 PPP 的访问控制与计费能力，以及以太网组网的经济性，具有适用范围广、安全性高、计费方便的特点，是 ADSL 等宽带接入的主流协议。

### 应用环境

- **DSL（Digital Subscriber Line，数字用户线路）**：以电话线为传输介质的宽带技术，泛称 xDSL。用户侧安装调制解调器，通过电话线接入 **DSLAM**（DSL 接入复用器，局端设备）。
- **DSLAM** 通过高速 ATM 或以太网把用户流量转发给 **BRAS**（宽带远程接入服务器，位于骨干网边缘）。

### 报文封装

PPPoE 报文用 Ethernet 格式封装。以太网头中 `Type` 字段：

- `0x8863`：承载 PPPoE **发现阶段**报文。
- `0x8864`：承载 PPPoE **会话阶段**报文。

PPPoE 头部字段：VER（版本 `0x01`）、Type（`0x01`）、Code（报文类型）、Session ID（会话 ID）、Length（Payload 长度）。

### 三个阶段

| 阶段 | 描述 |
| --- | --- |
| 发现阶段 | 获取对方以太网地址，确定唯一的 PPPoE 会话 |
| 会话阶段 | 包含 PPP 协商阶段和 PPP 报文传输阶段 |
| 会话终结阶段 | 会话建立后的任意时刻发送报文结束会话 |

### 五种发现阶段报文

| 类型 | 描述 |
| --- | --- |
| PADI | 发现初始报文，用户主机广播探测 PPPoE 服务器 |
| PADO | 发现提供报文，服务器回应 PADI，目的为客户端 MAC |
| PADR | 发现请求报文，主机单播请求选定的服务器 |
| PADS | 发现会话确认报文，服务器分配唯一会话 ID 并发给主机 |
| PADT | 发现终止报文，用户或服务器终止会话时发送 |

### 配置要点

PPPoE 客户端配置包含三个步骤：

1. **配置拨号接口**：

```text
dialer-rule                          # 进入 Dialer-rule 视图，配置发起会话的条件
interface dialer number              # 创建并进入 Dialer 接口
dialer user user-name                # 配置对端用户名（须与服务器 PPP 用户名相同）
dialer-group group-number            # 将接口置于一个拨号访问组
dialer bundle number                 # 指定 Dialer 接口使用的 Dialer bundle
```

2. **将 Dialer Bundle 和物理接口绑定**：

```text
pppoe-client dial-bundle-number number   # 绑定物理接口与 Dialer Bundle
```

ARG3 系列支持报文触发的按需拨号（on-demand）：物理线路 Up 后不立即拨号，有数据需要传送时才发起 PPPoE 呼叫。

3. **配置一条缺省静态路由**，使路由表中无匹配的流量都能通过拨号接口发起 PPPoE 会话。

验证命令：

```text
display interface dialer [number]            # 查看拨号接口配置，LCP opened、IPCP opened 表示链路正常
display pppoe-client session summary         # 查看会话状态与统计
```

会话 State 有四种：`IDLE`（空闲）、`PADI`（已发 PADI）、`PADR`（已发 PADR）、`UP`（建立成功）。

{{< callout type="warning" >}}
**MTU 问题**：以太网默认最大支持 1500 字节有效载荷。PPPoE 头部 6 字节 + PPP 协议 ID 2 字节，所以 PPPoE 帧的 MTU **不能超过 1492 字节**。
{{< /callout >}}

## BFD

**BFD（Bidirectional Forwarding Detection，双向转发检测）**是一个用于检测两个转发点之间故障的网络协议（RFC 5880），可提供**毫秒级**检测，实现链路快速检测。通过与上层路由协议联动，可加速路由收敛。

报文封装：BFD Echo 报文采用 UDP 封装，目的端口号 **3785**，源端口在 49152~65535 范围内。

### 为什么需要 BFD

现有故障检测方法各有不足：

- **硬件检测**（如 SDH 告警）：发现故障快，但并非所有介质都支持。
- **慢 Hello 机制**（路由协议的 Hello）：检测时间秒级，对高速传输和时延敏感业务而言太慢，且依赖路由协议。

BFD 提供了一种与介质和上层协议无关的快速故障检测机制，优点：

- 对两节点间链路（物理或逻辑，如 LSP、隧道）进行双向故障检测。
- 为不同上层应用（MPLS、OSPF、IS-IS 等）提供统一的故障检测服务和相同的检测时间。
- 故障检测时间远小于 1 秒，加速网络收敛，提高可靠性与服务质量。

### 工作过程

1. 依靠上层协议在两端点的链路上建立 **BFD 会话**（如 OSPF 邻居建立时把邻居信息告知 BFD）；多条链路可各建一个会话。
2. 在会话两端进行 BFD 检测，发现故障就拆除 BFD 邻居并立即通知上层协议，触发相应切换。

会话建立模式：

- **主动模式**：建立会话前主动发送 BFD 控制报文。
- **被动模式**：建立前不主动发送，直到收到对端报文。

> 通信双方至少要有一个运行在主动模式才能成功建立会话。

会话建立后的操作模式：

- **异步模式**：双方周期性发送控制报文，检测时间内没收到对端报文则认为链路故障。
- **查询模式**：假定每个系统有独立的连通确认方法，会话建立后停止发送控制报文，除非需显式验证连通性。

此外，BFD 还有**回声功能**：一端周期性发送 BFD 回声报文，对端不处理只转发回来，发送端据此检测会话状态。回声报文只检测直连网段，而控制报文还能检测非直连网段。
